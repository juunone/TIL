## Squash merge?

깃에서 머지의 종류는 크게 세가지로 나뉜다.  
머지, 리베이스, 스쿼시 이렇게 세가지로 볼수 있다.  

머지는 말그대로 커밋들을 일반 머지하는 부분이다.  
리베이스 머지는 base 자체를 `rebase` 키워드처럼 옮겨준다.  

예를들어, `develop` 브랜치에서 `feature` 브랜치를 딴후, 
`feature` 브랜치에서 한창 개발을 진행중일때, 여러명이서 협업하는 경우
`develop` 브랜치에 이번에 릴리즈에 같이 배포될 새로운 피처가 병합된 상태인경우 지금 작업중인 feature와 연관이있을경우
해당 커밋이 필요한경우가 있을 수도 있다.

이때 `{feature/branch} git rebase develop` 커맨드라인을
입력하면 기존에 a커밋마지막에서 따온 base가 마지막 피처가 머지된 b커밋 기준으로 base 기준이 변경된다.  

이렇게되면 커밋히스토리도 더 간단하게 가져갈수있어서, 추후에
다른 사람이 커밋히스토리를 확인 할 경우 명료하게 파악이 가능하다.  

지금까지 rebase 얘기를 했다면, 지금부터는 `squash merge`를 사용한 경험을 공유하겠다.  

rebase merge를 할경우 origin 까지 올라간 피처브랜치에
커밋이 여러개 존재할경우 혹시라도 `conflict`이 날경우
`rebase --continue` CLI를 통해 많은 커밋들의 컨플릭을 해결해줘야 하는 상황이 발생할 수 있다.  

이런 번거로운 작업을할때 한번에 해결가능한게 `squash` 다.  
`{develop} git merge --squash feature/branch`를 할경우 디벨롭에서 피처브랜치의 모든 커밋들을 **1개** 커밋으로
모두 합쳐준다. 물론 커밋메세지도 모두 합쳐진다.  

그럼 이때 컨플릭나는 부분을 모두 해결하고 커밋 머지해주면 여러개 커밋을 확인할 필요없이 한번에 가능하다.
